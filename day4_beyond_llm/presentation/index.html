Day 4: RAG & Agentic Systems
The Challenge

What limits traditional LLMs?

Vanilla Large Language Models are highly capable, but face severe architectural boundaries:

  • Fixed Knowledge: Frozen parameters restrict awareness to training cutoff dates.
  • Isolated Data: Cannot naturally read private company docs, PDFs, or secure databases.
  • Hallucination: Confidently invents plausible-sounding answers when lacking factual context.
  • Actionless Brain: Cannot execute actions or make API decisions autonomously.
  • Fleeting Focus: Limit to context window memory makes long-range workflows difficult.

🚨 The Scenario

"Summarize my company policy PDF"

Normal LLM Output:
❌ "I have never seen this PDF. Let me guess the policy based on standard guidelines..." (Generates fake answers!)
Part 1: RAG

Why RAG is the Solution

Instead of relying on pure "guessing", Retrieval-Augmented Generation grounds the LLM using a "Search + Answer" flow.

🔍 1. Retrieve

Reads local files (PDFs, DOCX, Databases) and fetches chunks containing the exact facts matching the user's query.

⚙️ 2. Augment

Stitches the retrieved raw text documents directly into the LLM prompt, providing real-time grounding facts.

✍️ 3. Generate

The LLM synthesizes a 100% grounded response, directly backed by the retrieved external knowledge sources.

RAG Architecture

The Semantic Retrieval Pipeline

User Query
↓
Embedding Model (Text Vectorization)
↓
Vector Search (FAISS Index Lookup)
↓
Relevant Chunks Retrieved & Loaded
↓
LLM Generates Grounded Answer

How it fits together:

Vector databases index document meanings. When a user asks a question, the vector similarity matching locates the exact chunk of text needed in milliseconds.

How RAG Works

Step-by-Step Implementation

01 Load Documents Incorporate external files: PDFs, DOCX, corporate database records, or wiki files.
02 Document Chunking Split documents into bite-sized, overlapping pieces (e.g. Chunk 1: Intro, Chunk 2: Settings).
03 Generate Embeddings Encode text chunks into high-dimensional vectors representing semantic meaning.
04 Store in FAISS Store vectors in index structures optimized for fast spatial similarity lookups.
05 Query Retrieval Vectorize user queries and fetch the top matching vectors (semantic search).
06 LLM Grounding Combine retrieved contexts + user query into a secure prompt to synthesize answers.
Part 2: Agents

Why Agentic AI Is Needed

While RAG connects LLMs to text search, **Agentic AI** allows the model to act as a dynamic brain, executing multi-step workflows and using tools.

📋 Simple Example:

“Read emails → extract invoice details → update the sales database → send a Slack notification”

This requires logical decision loops, integrations, and tool calls. That is where agents excel.

Agentic Autonomous Loops
Agent Architecture

Core Components of an AI Agent

🧠 1. Planning

Decomposes complex problems. ReAct frameworks allow agents to write thoughts, execute tools, evaluate results, and update their plans.

💾 2. Memory

Short-term: Interactive chat session state.
Long-term: Storing past executions in vector databases for historical recall.

🛠️ 3. Tools

Equips the LLM with superpowers: Database SQL runners, Email APIs, Web Search engines, Code interpreters, and CLI executors.

Agent Lifecycle

Typical Agent Execution Loop

User Request (High-Level Goal)
↓
Planner (Task Breakdown)
↓
Tool Selection & Execution
↓
Memory & Evaluation Update
↓
Final Grounded Response

Self-Critique & Execution:

Unlike sequential scripts, agents can observe if a tool execution failed (e.g. database syntax error), rewrite their query, and try again until the goal is achieved.

Decision Matrix

Use RAG vs. Agentic AI?

💡 Use RAG When

  • You have rich text files (PDFs, Wikis)
  • Data gets updated frequently
  • You need to search across large enterprise files
  • Preventing hallucination is highly critical
Examples: HR handbook bot, Tech support FAQ.

💡 Use Agents When

  • Multi-step workflows are required
  • Logic decisions are needed at runtime
  • Integration with local files / APIs is necessary
  • Tasks require planning and self-critique loops
Examples: DevOps pipelines, Autonomous writing bots.
Limits & Warning

When NOT to deploy them?

❌ Avoid RAG When:

  • The data is small and static (Simple prompting is enough)
  • Very low latency is needed (Retrieval + Embeddings add lag)
  • Database changes rarely (Index overhead is unnecessary)

❌ Avoid Agents When:

  • Tasks are single-step (Translate, summarize)
  • Workflows are strictly deterministic (Normal code is better)
  • High-risk critical systems (Medical control, Banking triggers)
⚠️ Risk Warning: Agents can misuse tools or execute unexpected code loops. Always build secure sandbox runtimes!
