## Section B - Theory Questions (50 Marks)

**Q1. Side Effects:**
*   **a) Why discouraged:** Streams are intended to be declarative and highly functional. Side effects violate referential transparency, meaning calling the operation multiple times might yield different results or perform hidden actions (like printing to console, or modifying external state), making code unpredictable and difficult to debug.
*   **b) Example/Alternative:**
    *   *Bad Example:* Using `forEach(s -> System.out.println(s))` (side effect: printing).
    *   *Better Alternative:* If the goal is just to calculate a sum, use `reduce(0, Integer::sum)` or `collect(Collectors.summingInt(Integer::intValue))`. This keeps the process pure and focused on data transformation.

**Q2. Parallel vs. Sequential Streams:**
*   **a) ForkJoinPool:** It is a thread pool managed by the Java Runtime Environment, optimized for tasks that can be easily divided into smaller, independent subtasks (divide and conquer). It efficiently manages worker threads, minimizing overhead and ensuring optimal load balancing across multiple CPU cores.
*   **b) Scenarios:**
    *   *Improvement:* Large datasets with independent processing logic (e.g., filtering millions of records, performing independent mathematical calculations).
    *   *Degradation:* Tasks with high synchronization overhead (locking/thread contention), or tasks that are inherently sequential (data dependency, i.e., processing a calculation step-by-step).

**Q3. `map` vs. `flatMap`:**
*   **a) `map` Failure:** The `map` operation is designed to transform an element $T$ into a single element $R$. If we apply `map(User::getSkills)`, the result is a `Stream>`. The output stream contains lists, not individual skills.
*   **b) `flatMap` Solution:** `flatMap` is required because the transformation function produces a *stream* (or collection). `flatMap` processes each `List` and then "flattens" (concatenates) all these individual streams together into a single, uniform `Stream`.

**Q4. Custom Collector:**
*   **Goal:** Aggregate a stream of `Integer` into a unique String (e.g., "1, 2, 3").
*   **Accumulator (`A`):** `A` represents the mutable reduction container (the running result). This method takes the current container (`A`) and the next element (`T`) and modifies the container to incorporate the new element (e.g., appending the new number and a comma to the running string).
*   **Combiner (`A`):** Used when running in parallel. It takes two independent, partially accumulated results (`A` and `A`) and merges them into a single container (`A`). This ensures that parallel threads can safely combine their partial results.
*   **Finisher (`R`):** Converts the final, mutable container (`A`) into the final, immutable result type (`R`) required by the user (e.g., trimming the final comma from the internal string buffer).

**Q5. Handling Empty/Null Streams:**
*   **a) Optional Role:** `Optional` is a container that may or may not hold a non-null value. It forces the developer to explicitly acknowledge the presence or absence of data, thereby making the code safer and reducing the risk of `NullPointerException`.
*   **b) Graceful Handling:** The pipeline should use `Optional.map()` strategically. If an intermediate operation might return an `Optional` (or a structure containing it), mapping the result prevents the pipeline from continuing if the optional is empty. Example: Instead of `stream.map(findData()).filter(Optional::isPresent).map(Optional::get)...`, one might use `stream.flatMap(optionalData -> optionalData.map(value -> processedValue).stream())`.
